## Introduction
본 과제는 현대적인 모방 학습(Imitation Learning) 파이프라인의 중요한 부분들을 직접 경험해 볼 수 있도록 안내합니다. 여러분은 SO-101 로봇 팔을 원격 조작(Teleoperation)하고, 전문가 행동을 모방하도록 정책(Policy)을 학습시켜 평가 과정에서 높은 성공률을 달성해야 합니다. 이번 과제를 위해 상당히 큰 규모의 코드베이스를 제공해 드립니다. 코드를 자유롭게 읽어보셔도 좋지만, 실제로 수정해야 하는 부분은 지정된 일부 소스 코드에 한정됩니다. 본 과제를 통해 이 분야의 현대적인 연구 및 산업계의 워크플로우가 어떻게 이루어지는지 좋은 영감을 얻으실 수 있기를 바랍니다.

## Tasks

### Setup
본 과제에서는 시뮬레이션 환경에서 SO101 로봇 팔을 원격 조작해야 합니다. 이번 과제의 원격 조작 설정을 올바르게 완료하려면 `python scripts/configure_keys.py`를 실행하십시오.
이 스크립트는 향후 원격 조작 시 사용할 키보드 키를 지정하도록 안내합니다. 각 방향 및 옵션에 권장하는 키가 제안되지만, 대부분 원하는 키를 자유롭게 선택하여 지정할 수 있습니다. 스크립트 실행이 완료되면 실습 전반에 걸쳐 사용될 `keymap.json` 파일이 생성됩니다. 저희는 MacOS 및 Ubuntu 환경의 스위스/미국식 키보드에서 이를 테스트했습니다. 키 지정 과정에서 문제가 발생할 경우, 문제를 해결하기 위해 `configure_keys.py` 스크립트를 수정하거나 `keymap.json` 파일을 직접 편집하셔도 좋습니다. 이 단계의 중요한 결과물은 실습 전반에 걸쳐 사용할 수 있는 `keymap.json` 파일을 확보하는 것입니다.

### Components
일반적인 모방 학습 스택의 많은 구성 요소들을 다루기 때문에(대부분은 이미 구현되어 제공됩니다), 우리가 사용하는 구성 요소들에 대해 간략히 설명합니다:
1. 데이터 저장(Data storage): 데이터 관리를 위해 널리 사용되며 다재다능한 선택지인 zarr (https://zarr.readthedocs.io/en/stable/)을 사용하여 데이터를 저장합니다.
2. 상태(States): zarr 데이터셋은 시뮬레이션의 원시 관측치(Raw observation)를 저장합니다. 학습을 위해 `--state-keys`를 지정할 때 아래의 키들을 사용할 수 있습니다 (슬라이싱(Slicing) 옵션 사용 가능, 예: `"state_cube[:3]"`):

   | Key | Dim | Description |
   |-----|-----|-------------|
   | `state_ee_xyz` | 3 | 말단 효과기(End-effector)의 데카르트 좌표계(Cartesian) 위치 (x, y, z). |
   | `state_ee_full` | 7 | 전체 말단 효과기 포즈(Pose): 위치 (3) + wxyz 형식의 방향 쿼터니언(Quaternion) (4). |
   | `state_joints` | 6 → 5\* | 6개 관절 모두의 관절 각도 (Rotation, Pitch, Elbow, Wrist_Pitch, Wrist_Roll, Jaw). \*평가 중에는 Jaw 관절이 자동으로 제외되어 5개의 제어 가능한 관절만 남습니다. |
   | `state_gripper` | 1 | 현재 그리퍼(Gripper) (Jaw)의 벌어짐 각도. |
   | `state_cube` | 7 | 큐브의 자유 관절(Free-joint) 상태: 위치 (3) + wxyz 형식의 방향 쿼터니언 (4). |
   | `state_obstacle` | 3 | 장애물 물체의 위치 (x, y, z). 장애물이 없을 때는 영벡터(Zero vector)가 됩니다. |
   | `goal_pos` | 3 | 보관함(Bin)의 중심 위치 (x, y, z). 실습 3과 같이 보관함 위치가 무작위로 변경될 때 유용합니다. |

   **Multicube 전용 키** (`--multicube` 옵션으로 녹화할 때만 사용 가능):

   | Key | Dim | Description |
   |-----|-----|-------------|
   | `original_pos_cube_red` | 7 | 빨간색 큐브 상태: 위치 (3) + wxyz 쿼터니언 (4). |
   | `original_pos_cube_green` | 7 | 초록색 큐브 상태: 위치 (3) + wxyz 쿼터니언 (4). |
   | `original_pos_cube_blue` | 7 | 파란색 큐브 상태: 위치 (3) + wxyz 쿼터니언 (4). |
   | `state_goal` | 3 | 목표 큐브 색상의 원-핫 인코딩(One-hot encoding) \[red, green, blue\]. |

3. 행동(Actions): 행동은 $a_t = s_{t+1} - s_t$와 같이 상태 간의 차이(Delta)로 계산합니다. 전체 말단 효과기 제어, xyz 전용 말단 효과기 제어, 그리고 관절 각도 제어가 이미 구현되어 있습니다. 더 자세한 정보는 `compute_actions.py`를 참고하시기 바랍니다.


### Exercise 1: MSE policy
여러분의 과제는 모방 학습을 사용하여 큐브를 집어 올리고, 장애물을 피해 이동한 후, 보관함 안에 넣을 수 있는 정책을 학습시키는 것입니다. 이는 두 단계의 과정으로 진행됩니다: 

#### 데이터 수집 및 준비 (Data collection and preparation)
이 실습의 데이터 수집을 위해 `record_teleop_demos.py` 스크립트를 사용해야 합니다. 이번 실습에서는 스크립트 플래그를 변경할 필요가 없습니다. 약 70%의 성공률에 도달하기 위해 일반적으로 15-20 에피소드 분량 이하의 데이터만 수집해도 충분할 것입니다. 이는 여러분의 모델에 따라 달라질 수 있습니다.
정책 학습을 위한 데이터를 준비하려면 `compute_actions.py` 스크립트를 실행해야 합니다. 이 스크립트는 정책의 행동 공간(Action space)을 선택하도록 요구합니다. 과제 2에서 로봇 팔의 관절 각도를 직접 제어하는 것과 말단 효과기 위치를 제어하고 역운동학(IK, Inverse Kinematics)을 수행하는 것의 차이를 경험하셨을 것입니다. 추가적으로, 학습 문제를 가능한 한 단순화하는 것이 권장됩니다. `compute_actions.py` 스크립트를 살펴보고 원하는 행동 공간을 선택해 보시기 바랍니다. 원격 조작의 난이도 차이는 있지만(대략적으로: 행동 공간이 복잡할수록 더 많은 원격 조작 노력이 필요함), 모든 행동 공간 선택지에서 실습을 완료할 수 있습니다.

**원격 조작 제어법 (Teleoperation controls)**
원격 조작 스크립트를 실행할 때, 로봇 팔의 말단 효과기 위치 제어(`configure_keys.py`에서 지정한 키 사용) 외에 다음과 같은 제어 기능을 사용할 수 있습니다:

| 동작 | 권장 키 | 설명 |
|--------|----------------|-------------|
| `record` | Space | 녹화 시작/정지 토글. 한 번 누르면 에피소드 녹화를 시작하고, 다시 누르면 일시 정지합니다. |
| `end_episode` | Enter | 현재 녹화된 에피소드를 종료 및 저장하고, 다음 에피소드를 위해 환경을 초기화(Reset)합니다. |
| `reset` | R | (녹화 중인 경우) 현재 에피소드를 버리고 환경을 초기화합니다. 원격 조작 중 실수를 했을 때 사용하십시오. |
| `escape` | ESC | 진행 중인 에피소드를 저장하고 세션을 종료합니다. |

화면 오버레이에는 현재 상태(`REC` / `IDLE`), 녹화된 에피소드 수, 그리고 녹화, 에피소드 종료, 초기화, 종료를 위한 키 바인딩 리마인더가 표시됩니다. `escape`를 누르면 녹화된 여러 에피소드가 하나의 `.zarr` 파일에 저장됩니다. 원격 조작 흐름은 다음과 같아야 합니다: `record` 누르기 -> 로봇 팔을 움직여 큐브를 집어 보관함에 넣기 -> `end_episode`를 눌러 해당 에피소드를 저장하고 환경 초기화하기.

또한 데이터가 기록될 때마다 증가하는 스텝 카운터(Step counter)를 볼 수 있는데, 이는 여러분이 로봇 팔을 제어하는 속도보다 빠르게 증가할 것입니다. 이 설정은 실제 로봇 원격 조작 환경을 모방한 것입니다. 이 스텝 카운터와 관련하여 명심해야 할 유일한 점은, 원격 조작자로서 녹화 중에 수 초 동안 아무것도 하지 않더라도 데이터는 계속 기록된다는 것입니다. 수 초 동안 가만히 서서 해당 데이터를 기록하고 이를 학습시키면, 일부 상태에서의 데이터 분포가 아무것도 하지 않는 방향으로 편향되기 때문에 대개 좋지 않은 정책으로 이어집니다. 일반적인 원격 조작 속도에서는 이 환경에서 무시할 수 있는 수준입니다. 데이터가 걱정된다면 간단한 플로팅(Plotting) 스크립트를 작성하여 검사해 보십시오.

**원격 조작 뷰 (Teleoperation views)**
카메라 시점을 다르게 설정하고 싶다면 변경하셔도 좋습니다. 이는 `mujoco` 파일에서 쉽게 변경할 수 있으며, 변경 여부는 여러분의 선택에 맡깁니다. 저희의 경우 현재 구현된 카메라 뷰를 사용하는 것만으로도 충분했습니다.

#### 원격 조작 방법 (How to teleoperate)
`configure_keys.py`에서 설정한 키를 사용하여 로봇을 움직일 수 있습니다. 녹화 키를 설정하셨을 것입니다 (스페이스 키 사용을 권장합니다). 에피소드를 녹화하고자 할 때 이 키를 반드시 누르십시오. 잘못된 움직임을 감지한 경우, 초기화 키(R 권장)를 사용하여 에피소드를 버리고 환경을 초기화할 수 있습니다. 에피소드가 끝나면 다음 에피소드로 넘어가기 위해 설정한 키(Enter 권장)를 누르면 됩니다. 원격 조작 시, 어떤 작업도 말단 효과기를 임의의 축으로 회전시킬 필요가 없다는 점을 참고하시면 유리할 것입니다.

#### 정책 학습 (Policy training)
정책을 학습시키려면 다음 TODO들을 완료해야 합니다:
- `train.py`에 있는 모든 TODO를 구현하십시오. 이 파일은 학습 실행 스크립트이므로, 편의를 위해 CLI 플래그를 추가하는 등 비교적 자유롭게 작성하셔도 좋습니다. 다만, 이미 강력한 기본 뼈대가 제공되어 있습니다.
- `model.py`에서 `ObstaclePolicy` 클래스의 TODO들을 구현하십시오.
- 정책 학습에 사용할 상태 및 행동 공간을 선택하십시오. `train.py` 스크립트를 실행할 때 CLI 인자로 이를 설정할 수 있습니다. 실습 1에서 정책을 학습할 때 장애물 상태(Obstacle state)를 반드시 포함해야 실습 2에서 해당 정책을 재사용할 수 있습니다. CLI 인자는 슬라이싱을 지원하므로 다음과 같이 실행할 수 있습니다:
```bash
python scripts/train.py ... --state-keys state_ee_xyz state_gripper "state_cube[:5]" --action-keys action_ee_xyz action_gripper --policy obstacle
```

#### 정책 성능 테스트 (Policy performance test)
`eval.py` 스크립트를 실행할 때 `--checkpoint` 플래그가 학습된 체크포인트를 가리키도록 설정하십시오. 정책의 출력을 시각적으로 검사할 수 있으며, `--headless` 모드로 실행하여 롤아웃(Rollout) 전반의 총 성공률을 확인할 수도 있습니다 (렌더링하는 것보다 훨씬 빠릅니다). `--num-episodes`를 사용하여 테스트할 에피소드 수를 늘릴 수도 있습니다.

모델 성능에 확신이 선다면 다음을 실행하여 성능을 확인할 수 있습니다:

참고: 이는 ARM MacOS 또는 Linux 시스템을 사용한다고 가정합니다. 다른 시스템을 사용하는 경우 WSL 또는 Docker를 사용하는 것을 권장합니다. `run_eval.py`는 평가용 유틸리티이므로 수정하지 않는 것이 좋습니다.

#### 참고 및 팁 (Notes and Tips)
- 정책의 대부분의 문제나 불량한 행동은 더 많은 데이터가 필요하거나 더 많은 파라미터가 필요하기 때문에 발생합니다. 실습 1과 실습 2에서 100% 성공률(SR)을 달성하는 데 1M 미만의 파라미터로도 충분합니다.
- 원격 조작 시 에피소드를 너무 일찍 수동으로 종료하지 않도록 하여 마지막 부분의 행동 청크(Action chunk)가 잘리지 않도록 하십시오.
- 정책 평가는 최대 800스텝 동안 실행됩니다. 이것이 정책의 한계 요인이 되지 않도록 하십시오.
- 이 실습 및 다른 모든 정책에서 큐브는 반드시 보관함 안으로 떨어져야 합니다 (성공 기준 z축 임계값에 도달했을 때 그리퍼 클로(Claw) 내부에 남아있으면 안 됩니다). 그리퍼 안에 큐브를 든 채 성공처럼 보이는 편법이 생기지 않도록 정책 동작을 직접 확인해 보세요.


### Exercise 2: DAgger
본 실습에서는 강의에서 소개된 DAgger(Dataset Aggregation) 알고리즘을 사용합니다. 이 알고리즘은 정책이 분포 외(OOD, Out of Distribution) 상태에 놓여 성능이 저하될 때 데이터를 추가하여 커버리지와 성공률을 높일 수 있게 해줍니다. 실습 1에서 학습시킨 것과 동일한 정책을 재사용합니다. `--adversarial` 플래그를 지정하여 `scripts/eval.py`를 다시 실행해 보면 성공률이 훨씬 낮아진 것을 확인할 수 있습니다. 그 이유는 장애물 분포가 변경되어 정책이 분포 외 상태로 강제되었기 때문입니다. 평가 중 정책 실행 과정을 관찰하고 정책의 실패 모드(Failure mode)를 이해해야 합니다. 그런 다음 `scripts/dagger_eval.py` 스크립트를 실행하면, 정책의 성능이 저조할 때 개입하여 분포 외 환경에서 더 많은 전문가 행동을 수집할 수 있습니다. 정책 실행 중에 개입하여 인간 전문가 행동을 제공하는 키는 이전의 녹화 키와 동일합니다. 이 작업의 나머지 키 매핑을 확인하려면 `dagger_eval.py` 파일을 살펴보시기 바랍니다 (예를 들어, 롤아웃 중 실패를 관찰했지만 개입하기에 너무 늦은 경우 해당 롤아웃을 초기화하고 더 일찍 개입할 수 있습니다). 그 후 분포 외 환경에서 더 많은 데이터를 수집하고 실습 1의 정책을 재학습시켜야 합니다. 정책 사양에 따라 성공률을 높일 수 있을 만큼 정책 커버리지가 충분히 커질 때까지 `실패 모드 관찰 -> dagger -> 행동 다시 계산 -> 재학습` 과정을 여러 번 반복해야 할 수도 있습니다. 만약 25개보다 훨씬 많은 dagger 에피소드를 수집해야 하는 상황이라면, 모델 크기와 에피소드 원격 조작 방식을 재고해 보는 것을 권장합니다.

#### 로컬 평가
모델 성능에 확신이 선다면 다음을 실행하여 성능을 확인할 수 있습니다:
```bash
python student_eval/run_eval.py --exercise 2 --checkpoint <path to your ckpt>
```
이 명령은 `ex2_result.hwresult` 파일을 생성합니다. 자기 점검용 결과 기록으로 보관할 수 있습니다.
가장 우수한 체크포인트 `.pt` 파일과 `model.py`를 함께 보관해 두세요. `ObstaclePolicy` 클래스 이름은 평가 스크립트에서 임포트하므로 변경하지 마십시오.

#### 팁 및 참고 (Tips and Notes)
- DAgger 중에 일반적으로 수행되는 것처럼, 여러분이 기록한 상태(및 행동)만 `.zarr` 파일에 저장됩니다. `compute_actions.py`를 실행하면 `datasets/raw/single_cube/teleop/`에 있는 모든 원격 조작 에피소드와 `datasets/raw/single_cube/dagger/`에 있는 새로운 dagger 에피소드를 가져와 하나의 처리된 `.zarr` 파일로 결합합니다.
- 화면에 표시되는 dagger 단계는 기본적으로 실습 1과 동일합니다. 크게 걱정할 필요는 없지만 수 초 동안 가만히 서 있지 않도록 노력하십시오.
- 이 실습이 얼마나 쉽게 느껴지는지에 따라, DAgger가 왜 매우 유용한 도구인지, 동시에 실제로는 왜 매우 자원 집약적인지 이해할 수도 있고 그렇지 못할 수도 있습니다.


### 실습 3: 멀티큐브 목표 조건부 모방 학습

본 실습에서는 동일한 정책을 사용하여 여러 작업을 해결할 수 있는 목표 조건부 정책(Goal-conditioned Policy)을 학습시킵니다. 단일 행동을 학습하는 대신, 로봇은 작업 정보를 입력으로 받아 그에 맞게 행동해야 합니다.

#### 작업 설명 (Task Description)

multicube 환경에는 세 개의 큐브가 있습니다:

* 빨간색 (Red)
* 초록색 (Green)
* 파란색 (Blue)

각 에피소드가 시작될 때:

* **큐브 위치가 무작위로 지정됩니다**
* **보관함 위치가 무작위로 지정됩니다**

에피소드에 대한 목표 큐브 색상이 지정됩니다. 로봇은 올바른 큐브를 집어 보관함에 넣어야 합니다. 이 정책은 목표 조건부이므로, 작업 목표를 설명하는 추가 입력을 받습니다. 이는 모방 학습에 관한 강의 3에서 다룬 내용을 적용하는 것입니다.

#### 데이터 수집 (Data Collection)
먼저, 평소와 같이 원격 조작을 사용하여 데모를 녹화합니다. 다음 추가 플래그가 필요합니다:

```bash
python scripts/record_teleop_demos.py --multicube
```

녹화하는 동안 다음과 같은 추가 목표 정보가 저장됩니다:

* `state_goal` — 목표 큐브 색상의 원-핫 인코딩
* `goal_pos` — 보관함 중심 위치

#### 모델 (Model)
여기서는 `MultiTaskPolicy` 아래에 새로운 모델을 자유롭게 구현할 수 있습니다. 이전과 동일한 정책을 선택하거나 완전히 새로운 정책을 구현할 수 있습니다. 추가 라이브러리를 임포트하지 않고 지나치게 큰 모델을 사용하지 않는 한 어떠한 제한도 두지 않습니다.


#### 학습 (Training)

multicube 데이터셋으로 학습할 때, 정책은 모든 목표 조건부 입력을 받아야 합니다.

다음 상태 키들을 포함하여 `train.py`를 실행하십시오:

- `original_pos_cube_red`
- `original_pos_cube_green`
- `original_pos_cube_blue`
- `state_goal`  
- `goal_pos`

학습 문제를 다시 단순화하기 위해 이전과 같이 임의의 슬라이싱 방법을 선택할 수 있습니다.

#### 평가 (Evaluation)

학습된 정책을 다음 명령으로 평가합니다:

```bash
python scripts/eval.py --checkpoint <path_to_checkpoint.pt> --multicube
```

#### 난이도 (Difficulty)

이 multicube 목표 조건부 문제는 이전 실습들보다 **현저히 어렵습니다**. 저희는 평가 성공률(SR)이 50%에 가까워지면 매우 우수한 성능으로 간주합니다. 다시 90%나 그와 유사하게 높은 성공률에 도달할 것으로 기대하지 않습니다. 또한, 이 시점에서 여러분은 모방 학습이 학습 데이터의 양에 따라 잘 확장(Scale)된다는 것을 알고 계실 것입니다. 단순히 성공률을 높이기 위해 원격 조작에 10시간을 허비하기보다는, 성공률을 향상시킬 수 있는 더 영리한 방법들을 고민해 보시기 바랍니다 (이미 몇 가지를 접해 보셨을 것입니다).
